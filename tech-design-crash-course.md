## Module 1 — The story (learn to tell it in 3 minutes)

Every good tech design opens with a story, and yours is genuinely good. Memorise the beats:

**Beat 1 — the problem.** Kedro loads and saves data all day and never checks any of it. Teams that want validation write *hooks*: a separate class in `hooks.py`, registered in `settings.py`, that intercepts datasets by name. Three problems: you can't tell from a node or a catalog entry that validation exists (hidden control flow); every new dataset means editing the hook; and — the killer — **dataset hooks only fire during `kedro run`**, so a notebook `catalog.load()` is completely unguarded.

**Beat 2 — the first attempt (KEP-7, April).** Mirror parameter validation: put Pandera schemas in node **type hints**, discover them by walking pipelines, wrap matching datasets in a `_ValidatingDataset` proxy. Prototype worked. Community review found the cracks: if two pipelines annotate the same dataset with different schemas, the **last one silently wins** (deepyaman); a bare `Schema` annotation **lies to type-checkers** — the runtime value is a DataFrame; and schemas living next to nodes create **cross-pipeline imports** and bloat (Adeikalam, from real production experience). The vote was paused.

**Beat 3 — the pivot (KEP-10, accepted +3).** Move the declaration to where the dataset itself is declared: a `validator:` key in `catalog.yml`. One dataset, one validator, *by construction* — the conflict bug isn't mitigated, it's impossible. Enforcement moves into the catalog's own `load()`/`save()` path (no wrapper), so it covers notebooks, IDE, CI — everything that goes through the catalog. And the backend is a tiny protocol — `validate(data) -> data` — so Pandera is just the reference implementation; GX, Pydantic, and custom non-tabular validators (Adeikalam validates *matplotlib plots*) all fit.

**Beat 4 — where we are.** KEP accepted. Working prototype on `prototype/kep7-v2-catalog-validation`: full test suite green, runnable demo. Today's session: walk the architecture, lock 7 implementation decisions, agree the PR plan.

## Module 2 — The concepts, taught inline

Five mini-lessons. Each teaches the general concept *and* lands it on the exact spot in this feature.

### 2.1 The DataCatalog (the thing you're modifying)

The catalog is Kedro's registry of named datasets. `catalog.yml` maps names → configs (`type`, `filepath`, …); `DataCatalog.from_config()` turns that into an object where `catalog.load("companies")` reads the CSV and `catalog.save("out", df)` writes it. Three internals matter tomorrow:

**a) One parsing funnel.** Every entry — whether written explicitly or produced by a *dataset factory pattern* like `"{partner}_shuttles":` — passes through `DataCatalog._add_from_config()` before a dataset exists. Factories resolve **lazily**: the pattern sits dormant until someone asks for `acme_shuttles`, at which point the resolver fills in `{partner}` and calls `_add_from_config` with the concrete config. **Why you care:** capture the `validator:` key there and you've covered explicit entries *and* factories with one hook point, for free. That's the answer to Elena's April concern about "wrapping datasets not used in this run" — specs are parsed (strings only!) at build; nothing heavier happens until the dataset is actually used.

**b) Lazy datasets and `__setitem__`.** Configs become `_LazyDataset` placeholders; the real dataset object is *materialised* on first access (`get()`). Materialisation pops the lazy entry **before** assigning the real one, so it does **not** trip `__setitem__`'s "replacing existing dataset" branch. Explicit replacement (`catalog["companies"] = df` in a notebook) **does** trip that branch, which clears related state. **Why you care:** the validator spec's lifecycle rides exactly this: *the spec follows the catalog entry, not the name forever*. Replace the entry → spec cleared (no stale schema firing at your notebook DataFrame). Materialise a factory → spec kept. Both are pinned by tests.

**c) `load()`/`save()` are the chokepoint.** Everything that reads/writes through the catalog goes through these two methods — the runner (`Task`), notebooks, `kedro ipython`, plugins. **Why you care:** the last line of `load()` is now `return self._maybe_validate(ds_name, dataset.load(), mode="load")`, and `save()` is `dataset.save(self._maybe_validate(ds_name, data, mode="save"))`. That's the whole integration: ~25 lines in `kedro/io/data_catalog.py`.

What deliberately does *not* go through the chokepoint: `catalog.get(name).load()` (raw dataset access) and Kedro-Viz `preview()`. That's by design — the programmatic API needs a raw path so it never double-validates — and it's stated openly in the KEP's risk register. Never claim "everything is covered"; claim "everything *through the catalog* is covered, and here's the documented bypass list."

### 2.2 Protocols and the footgun (the pluggability mechanism)

Python has two kinds of typing. *Nominal*: "you're a Duck if you inherit from Duck." *Structural*: "you're a Duck if you quack." `typing.Protocol` is structural typing made official — you declare the shape:

```python
@runtime_checkable
class Validator(Protocol):
    def validate(self, data: Any) -> Any: ...
```

Any object with a `validate` method satisfies this — no imports from Kedro, no base class. That's why a Pandera schema, a Great Expectations wrapper, and a 5-line metrics checker can all be validators without knowing about each other. `@runtime_checkable` additionally lets you test conformance at runtime with `isinstance(obj, Validator)`.

**The footgun** (know this cold — it's the sharpest technical moment you own): `isinstance` against a runtime-checkable protocol only checks *attribute presence*. And a **class object** `MetricsValidator` *has* an attribute called `validate` (the unbound method). So `isinstance(MetricsValidator, Validator)` — the class itself, not an instance — returns `True`. If `resolve_validator` duck-typed before checking `inspect.isclass`, a YAML entry like `class: pkg.MetricsValidator, options: {required_keys: [...]}` would return the **uninstantiated class**, silently drop the options, and explode with a confusing `TypeError` at first load. Hence the iron ordering in `resolve_validator` ([kedro/validation/core.py](kedro/validation/core.py)):

1. **Adapters first** (Pandera gets first refusal on the imported object)
2. **Classes are instantiated** with `options`, then the *instance* is protocol-checked — never the class object
3. **Non-class instances** are protocol-checked directly (options on an instance = config error, they can't be applied)
4. **Plain callables** get wrapped in `CallableValidator`
5. Anything else → `ValidationConfigurationError` naming the path and what was found

There's a regression test that deliberately reproduces the footgun. If anyone asks "why is the resolution order like that?" — this is the answer, and it lands well because it's a real bug you caught.

### 2.3 Pandera in 15 minutes (the reference backend)

Pandera validates DataFrames against schemas. Two styles: **`DataFrameModel`** (class-based, like the `CompaniesSchema` example — fields with checks like `ge=0`, `unique=True`) and **`DataFrameSchema`** (an object you build programmatically, or load from YAML with `from_yaml`). The adapter accepts both — which is why "YAML-defined schemas already work in v1" (a module-level `schema = DataFrameSchema.from_yaml(...)` referenced by dotted path).

Four behaviours you must know:

- **`lazy=True`** collects *all* failures instead of stopping at the first, raising `SchemaErrors` whose `failure_cases` attribute is itself a DataFrame of every failing (column, check, value, row). The adapter groups those by `(column, check)` into `CheckFailure` records with capped examples (5) — that's why a million-row failure prints as three readable lines. Full report survives on `err.__cause__`.
- **`coerce=True`** means the schema *transforms*: a float64 `id` column comes back int64. So validators may legitimately return **different data** than they received — the node gets the coerced frame on load, the disk gets it on save. This is a documented contract with a real consequence: `after_dataset_saved` hooks see the *pre*-coercion object (the runner holds its own reference), so what's persisted can differ from what save-hooks observed. Known, documented, tested.
- **The namespace split (pandera ≥0.24):** base `pandera` ships **no dataframe backend**. You install `pandera[pandas]`, `pandera[polars]`, etc., and import from `pandera.pandas` / `pandera.polars` / `pandera.pyspark`. The old top-level `pandera.DataFrameModel` is deprecated (0.30) and slated for removal. The adapter therefore detects via the *shared base class* all backends inherit from — detecting only the legacy top-level class (as the first prototype did) would miss every modern schema. This also drives tomorrow's **extras-naming decision** (per-backend `kedro[pandera-pandas]`, deepyaman's ask).
- **pyspark is weird:** `pandera.pyspark` **never raises** — errors accumulate on a `df.pandera.errors` accessor, and the backend lacks element-wise checks, sampling, and `drop_invalid_rows`. The adapter reads the accessor and raises so invalid Spark data can't silently pass — but the honest position (and deepyaman's review) is: keep the code, advertise pandas + Polars, revisit when pandera's Narwhals-backed unification (0.32.x, not yet on PyPI) matures. That's decision 3 tomorrow.

One more polars note: a `LazyFrame` validates at *schema-only depth* by default (data-level checks silently skipped) — the adapter warns once per dataset and offers `options: {collect: true}`.

### 2.4 Hooks and pluggy (why "just use a hook" fails)

Kedro's hook system is built on **pluggy** (pytest's plugin engine): the framework defines hook *specs* (`after_dataset_loaded`, etc.), plugins register *implementations*, and the framework calls them at set points. Two mechanical facts destroy the hooks alternative — memorise both citations:

1. **Where they fire.** `before/after_dataset_loaded/saved` are invoked only inside `Task` in the runner — *not* in `DataCatalog.load/save`. A notebook `catalog.load()` fires zero dataset hooks. Cite it as "**only in kedro/runner/task.py**": the sequential path is 151–188, but the async path also fires them (222–233, 247–251) — say "only in Task", never "only lines 151–188", or a panelist reading along corrects you. (And be precise the other way: *lifecycle* hooks like `after_catalog_created` genuinely do fire outside runs — the blanket claim "hooks run outside pipelines" is true for those and false for the dataset I/O hooks validation needs. Making this distinction unprompted is what earns trust.)
2. **Return values are discarded.** `Task` binds the loaded object *before* the hook fires (`inputs[name] = catalog.load(name)`, task.py:152), and pluggy throws hook return values away. A hook can *raise*, but it can never *replace* the data the node receives — so coercion, and therefore the whole `validate(data) -> data` protocol, is unimplementable as a hook.

And the hybrid steelman ("use an `after_catalog_created` hook to *wrap* the datasets — that fires outside runs!"): that's exactly the rejected v1 wrapper with a different trigger, inheriting every wrapper liability, plus a double-wrap hazard when plugins rebuild catalogs inside hooks. Name it before anyone proposes it.

The one honest hooks advantage: `Task` fires hooks regardless of catalog implementation, while the funnel only exists in `DataCatalog`. A bespoke `DATA_CATALOG_CLASS` loses validation — which is why the context logs a prominent warning in exactly that case (decision 7's design).

### 2.5 The supporting cast (five small things that make you look thorough)

- **The YAML "Norway problem."** YAML 1.1 parses unquoted `on`, `off`, `yes`, `no` as booleans. So `on: [save]` inside the validator block arrives as `{True: ["save"]}` — verified all the way through Kedro's `OmegaConfigLoader`. `ValidatorSpec.from_config` normalises the `True` key back to `"on"`, so the KEP's examples work as written. Bonus fact: this is latent in *all* Kedro YAML today (a parameter literally named `on` becomes `True` right now); our key is just the first core feature to walk into it. Great "we sweated the details" moment.
- **The sentinel.** `validate_catalog_dataset(catalog, name, data=...)` needs to distinguish "caller didn't pass data → load it raw" from "caller passed `None` → validate `None`" (a JSON metrics dataset can legitimately hold `None`). Defaulting to `None` can't express that, so an internal module-level `_UNSET = object()` marker is the default. Users never type it — they just omit the argument.
- **Pickling & ParallelRunner.** `ParallelRunner` ships the catalog to worker processes via pickle. Resolved validator objects may not pickle (and shouldn't need to), so `__getstate__` drops the resolved cache and the per-process `_save_validated` set; workers re-resolve lazily from the string specs. Requirement: schema modules importable in workers — already true for node functions. (This is also wrapper-failure #3: proxy objects had to survive `ForkingPickler`.)
- **`find_spec` pre-flight.** `importlib.util.find_spec("pandera")` asks the import machinery "does this package exist?" *without executing it* (caveat: on a dotted submodule it imports the parent — which is why only the top-level package is checked). At catalog build, every spec's top-level package is pre-flighted: a miss warns immediately instead of exploding hours into a run. Precedent inside Kedro core already: the optional-`rich` checks (`kedro/framework/project/__init__.py:257`, `kedro/ipython/__init__.py:53`) and the `settings.py` pre-flight (`project/__init__.py:368`) — direct answer to deepyaman's "is this pattern used elsewhere?" question.
- **The kill switch and layering.** `kedro.io` is not allowed to import framework settings (layering). So: the `DATASET_VALIDATION` setting is applied by `KedroContext` onto `catalog.validation_enabled`, while the emergency env var `KEDRO_DATASET_VALIDATION` is read *directly inside* `_maybe_validate` (precedent: `KEDRO_MP_CONTEXT` in the runner) and **wins over the setting in both directions**. Why that matters: plugins (kedro-telemetry) rebuild catalogs from config inside hooks — those instances never meet the context, but the env var still governs them. `DataCatalog` itself defaults `validation_enabled=True` so declared config is honoured everywhere by least surprise. And `validate_catalog_dataset` ignores the flags entirely — explicit calls always validate (noklam's IDE never needs the env var).

## Module 3 — The trace drill (your Frame 5 narration)

Learn to narrate this blind. Do it out loud, twice, with the diagram; once without.

> "A user writes `validator: pkg.CompaniesSchema` on a catalog entry. At catalog build, `_add_from_config` — the single funnel every entry passes through, factories included — captures that into a `ValidatorSpec`: pure strings, no imports, so a 500-entry catalog builds exactly as fast as before, and a cheap `find_spec` pre-flight warns if the backing package isn't installed.
>
> Now something calls `catalog.load('companies')` — the runner, a notebook, doesn't matter. The dataset loads *untouched* — no wrapper, real type, real repr. On the way out, the data passes through `_maybe_validate`, a guard ladder ordered cheapest-first: kill-switch off? return. No spec for this name? — that's a dict miss, the 99% path, effectively free. Wrong mode or disabled? return. Then, *first use only*, the validator resolves: import the object, offer it to the Pandera adapter, otherwise instantiate-and-protocol-check, cache the result.
>
> `validator.validate(data)` runs. Pass — the node receives the (possibly coerced) frame. Fail — the exception is enriched *by the funnel* with the dataset name and mode (the validator itself is dataset-agnostic — the same schema can serve `X_train` and `X_test`), and raised as a `DataValidationError` with failures grouped per check, capped examples, full backend report on `__cause__`. Unless `severity: warn` — then it logs the same report and passes the original data through.
>
> Save is symmetric, with one bonus: validation runs *before* `dataset.save()`, so invalid data never lands on disk."

Checkpoints hidden in that narration you should be able to expand on demand: why dict-miss = the performance answer · why enrichment lives in the funnel · why resolution is cached per catalog instance · why `severity: warn` returns the *original* data.

## Module 4 — The 7 design decisions (defend each in 30 seconds)

Format: **Decision → the rejected alternative → the two-sentence defense.** These are your Frame 6 cards; the board carries the text, you carry the *why*.

1. **Funnel, not wrapper.** *Rejected: `_ValidatingDataset` proxy (the April prototype — flag openly that this deviates from your own pivot announcement).* A proxy breaks `isinstance` in the catalog's version handling, must fake runner-checked flags (`_EPHEMERAL`, `_SINGLE_PROCESS`), and must survive pickling for ParallelRunner; deleting it also deleted deepyaman's repr complaint rather than patching it. The funnel touches none of that — ~25 benchmarked lines at the chokepoint.
2. **Schema-on-dataset, not on nodes.** *Rejected: KEP-7's type-hint discovery.* One YAML key per dataset makes conflicts structurally impossible and puts the declaration in the artifact teams already read to discover datasets; type hints lied to mypy and coupled pipelines (Adeikalam's production evidence). Node hints remain welcome as documentation.
3. **A protocol, not Pandera coupling.** `validate(data) -> data`, any raise == failure. Pandera fits in ~200 adapter lines against public API only; GX/Pydantic/plain-callable/non-tabular all satisfy the same shape — proven with worked sketches, including a metrics-dict validator with no pandera anywhere.
4. **Lazy resolution + eager pre-flight.** Specs are strings at build (zero startup cost, factory-compatible — the resolution of Elena's April timing concern); imports happen at first use; `find_spec` pre-flight + `--resolve-only` CLI + `"strict"` mode cover fail-fast needs at three escalating levels.
5. **Validate both directions; never silently skip.** Save-validation runs pre-write. Auto-skipping load-after-save would certify data the schema never saw (CSV round-trips aren't identity-preserving); the opt-in `skip_load_after_save` is scoped to "*this process* wrote it" — safe under partial runs, unlike `on: [save]`.
6. **A kill switch that reaches everything.** Setting for policy, env var for emergencies, env read inside `kedro.io` so plugin-rebuilt catalogs obey it; explicit API calls ignore both (one consistent rule for tooling).
7. **Nothing added to `CatalogProtocol`.** Adding members to a runtime-checkable protocol instantly fails settings-load for every existing custom catalog — a breaking change by side effect. Instead: `hasattr` check + loud warning when a custom catalog would silently ignore declared validators.
