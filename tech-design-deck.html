Frame 3 · The proposal in one screen5 min

One line in the catalog. Nodes stay plain Python.

The entire user experience is: write a schema in a conventional place, point one new catalog key at it. From then on, every load of that dataset is validated before the data reaches a node, and every save is validated before anything is written to disk.

newconf/base/catalog.yml — the one thing users learn

companies:
  type: pandas.CSVDataset
  filepath: data/01_raw/companies.csv
  validator: spaceflights.schemas.
      companies.CompaniesSchema

conventionsrc/spaceflights/schemas/companies.py

import pandera.pandas as pa

class CompaniesSchema(pa.DataFrameModel):
    id: int = pa.Field(nullable=False,
                       unique=True)
    company_rating: float = pa.Field(
                       ge=0, le=1)

unchangednodes.py — zero validation machinery

def preprocess_companies(
    companies: pd.DataFrame
) -> pd.DataFrame:
    # plain python, plain DataFrame
    ...

Load and save. Save-side validation runs before the write, so invalid data never lands on disk.

Everywhere, not just kedro run. Notebooks, the IDE extension, CI — anything going through the catalog.

One dataset, one validator, by construction. The v1 "last schema silently wins" bug is structurally impossible, not just mitigated.

Pluggable. The backend contract is validate(data) in, data out, raise on failure. Pandera is the reference; GX, Pydantic, custom and non-tabular checks all fit.

What failure looks like — grouped, bounded at any frame size, full Pandera report attached underneath:

DataValidationError: Validation failed for dataset 'companies' on load
3 check(s) failed — 4 failure case(s):
  - id: field_uniqueness           — 2 cases (e.g. 3888, 3888)
  - company_rating: str_matches    — 1 case  (e.g. not-a-rating)
  - iata_approved: isin(['t','f']) — 1 case  (e.g. x)
# names the dataset, the mode, every failed check with examples
Say"Note what the node looks like: completely unchanged. Validation is declared where the dataset is declared, and the error message is the pitch — every failed check at once, grouped with examples, at the I/O boundary, before any node ran. On a ten-million-row frame this error is still three readable lines; the full Pandera report stays attached underneath for debugging."

Frame 5 · Architecture — follow one load8 min + questions

From YAML to validated DataFrame

One diagram carries the whole design. Follow a single catalog.load("companies") from config to node, and notice two properties: build time costs nothing (strings only), and the dataset object itself is never touched — there is no wrapper anywhere.

conf/base/catalog.yml companies: validator: pkg.Schema — shorthand string, or a long form with on/severity/options.
▼
DataCatalog._add_from_config Captures the key into a ValidatorSpec — strings only, no imports, zero build cost. This is the single funnel every entry passes through, explicit and factory patterns, so patterns just work. The resolver's config store is never mutated (a cleaned copy goes to the dataset).
▼
catalog.load("companies") → dataset.load() The dataset is untouched: real class, real repr, picklable as-is. No wrapper.
▼
_maybe_validate(data, "load") — the funnel, a guard ladder ordered cheapest-first
  • 1 · kill switch (env var / flag) off? → return data — zero cost
  • 2 · no spec for this name (dict miss)? → return data — the 99% path
  • 3 · wrong mode, or per-dataset disabled? → return data
  • 4 · skip_load_after_save applies? → return data
  • 5 · resolve_validator(spec) — first use only: import, adapt, cache
▼
validator.validate(data) The Pandera adapter, or any object satisfying the protocol.
▼
valid The node receives the frame — possibly dtype-coerced, which is a documented contract.
invalid DataValidationError, enriched by the funnel with dataset name + mode (the schema itself is dataset-agnostic), failures grouped per check. With severity: warn it logs and passes the original data through.

Save is symmetric

dataset.save(_maybe_validate(data, "save")) — validation runs before the write, so invalid data never lands on disk.

Evidence for "why not hooks", if asked

  • Dataset I/O hooks fire only in kedro/runner/task.py — a notebook catalog.load() fires none. The funnel covers both.
  • pluggy discards hook return values — a hook can raise, but can never hand the node coerced data, so the protocol is unimplementable as a hook.
  • _add_from_config being the single config funnel is what makes factory patterns work with no extra machinery.

Timing, for the April question

Nothing heavier than string parsing happens at catalog build. Imports and validation happen at first use of the dataset — so building a 500-entry catalog costs the same as before, and datasets a run never touches never pay anything.

Say"Two things to take from this diagram. First, the cost story is structural: a dataset without a validator pays one dictionary miss per load, and build time is strings only. Second, the enrichment step: the schema is deliberately dataset-agnostic so one schema can guard X_train and X_test — it's the funnel that stamps the dataset name and mode onto the error. This is the right frame for mechanics questions."
Frame 6 · Design decisions already made4 min

Seven decisions locked in the accepted KEP — one flagged openly

These are settled by the vote; they're here so the reasoning is visible. The one to be upfront about is the first — it deviates from what I announced in the pivot message.

1 · Funnel, not wrapper

Deviation, flagged: I announced the wrapper would carry over. Building it proved otherwise — it breaks isinstance in version handling, fakes runner flags, breaks pickling under ParallelRunner, and pollutes the repr (the original review complaint). The funnel is ~25 lines at the chokepoint, benchmarked.

2 · Schema-on-dataset

One YAML key per dataset makes conflicts impossible and puts the declaration in the file teams already read to discover datasets. The schemas/ folder is convention, not enforcement — any importable dotted path works; no auto-detection because the explicit path is the discovery.

3 · A protocol, not Pandera coupling

validate(data) in, data out, any raise counts as failure. Pandera fits in ~200 adapter lines against public API only. GX, Pydantic, plain functions and non-tabular checks satisfy the same shape.

4 · Lazy resolution + cheap pre-flight

Specs are strings at build; imports happen at first use. A find_spec pre-flight warns about missing packages at build (same pattern kedro already uses for optional rich), and kedro catalog validate --resolve-only is the CI gate.

5 · Both directions, no silent skip

Save validation runs pre-write. Auto-skipping load-after-save would certify data the schema never saw (CSV round-trips drop dtypes). The opt-in skip_load_after_save is scoped to "this process wrote it", which stays safe under partial runs.

6 · A kill switch that reaches everything

DATASET_VALIDATION setting for policy; KEDRO_DATASET_VALIDATION env var for emergencies, read inside kedro.io so it governs every catalog instance — including ones plugins rebuild (the telemetry case). Explicit API calls always validate.

7 · Nothing added to CatalogProtocol

Adding members to a runtime-checkable protocol instantly fails settings-load for every existing custom catalog. Instead, the context warns loudly when a custom catalog would silently ignore declared validators.

The through-line

Declare validation where the thing itself is declared. Parameters on node signatures (KEP-1), datasets on catalog entries (this KEP). And the runtime object stays what it naturally is — nodes get DataFrames, not model instances.

Frame 7 · Why not hooks / why not the wrapper3 min

The comparison in one table

HooksWrapper (v1 prototype)Funnel (chosen)
Covers notebooks / IDE / CINo — dataset hooks fire only inside the runner's TaskYesYes
Can coerce — validate(data) returns dataNo — pluggy discards hook return valuesYesYes
Dataset repr and isinstance stay honestYesNoYes
ParallelRunner picklingYesFragileYes
after_dataset_loaded sees validated data deterministicallyNo — depends on plugin registration orderYesYes
Core surfaceNoneMedium~25 LOC in kedro/io; no-op benchmark ships with PR 1

The one honest hooks advantage: Task fires hooks regardless of the catalog class, while the funnel exists only in DataCatalog. A fully custom DATA_CATALOG_CLASS loses declared validation — which is exactly why decision 7's loud warning exists. And hook users lose nothing: hooks keep firing in the same places, now with a stronger guarantee (they always observe validated data on load).

Say"The hooks column isn't opinion — both failures are mechanical. Dataset hooks are invoked only by the runner's Task, so a notebook load fires none. And pluggy throws hook return values away, so a hook can never replace the data a node receives — which makes coercion, and the whole protocol, unimplementable as a hook. I'd rather concede the one real hooks advantage openly: it's the custom-catalog case, and we warn loudly there."

Decisions 1–4

For each: the question, the options, my recommendation, and space to record what we lock. If one stalls past four minutes: both positions on the card, an owner and a 48-hour async deadline, and we move on.

1 · Extras naming

A: kedro[validation] = pandera[pandas]  ·  B: per-backend kedro[pandera-pandas] / [pandera-polars] / [pandera-pyspark], thin 1:1 forwarders to pandera's own extras.

Recommend B  Since pandera 0.24 the base package ships no dataframe backend, so a generic extra silently picks one for the user. B mirrors the kedro-datasets ibis convention. deepyaman and Ravi are both on record for named-backend extras. Owning the gap: the branch still ships kedro[validation] plus two install-hint strings — the sweep is a named PR-1 item.

2 · Public API name

A: validate_catalog_dataset  ·  B: validate_dataset.

Recommend B  The catalog is already the first argument, so the prefix is redundant; it pairs cleanly with the batch validate_catalog; and the primary consumer (the VSCode extension) prefers it. Nothing has shipped — the rename is free today and a deprecation cycle later.

DECIDED:
3 · pyspark support

A: ship the pyspark branch as experimental  ·  B: keep it out of the advertised support matrix until the Narwhals-backed pandera backend ships (0.32.x, not yet on PyPI).

Recommend: keep the code, advertise B  Pandera's pyspark backend never raises — errors accumulate on an accessor — and it lacks element-wise checks and sampling. Our adapter reads the accessor so invalid Spark data cannot silently pass, but the marketed matrix is pandas + Polars.

Frame 9b · Decisions to lock today (5–7)8 min

Decisions 5–7

5 · ContextAwareValidator

A: ship a richer validate(data, context) protocol variant in v1  ·  B: reserve the name, ship later.

Recommend B  The funnel already stamps dataset name and mode onto every error, so the main use case is covered. Widening a protocol before a single third party has implemented it is premature.

DECIDED:
6 · CLI default mode

A: kedro catalog validate runs full data validation by default  ·  B: resolve-only by default; --all-data is explicit.

Recommend B  Unbounded data loads should never be a CI default. Resolve-only catches typo'd paths and missing dependencies in milliseconds — which is also the fail-fast answer for lazy resolution. Sub-decision: severity: warn findings exit 0 by default; --strict-warnings flips that.

DECIDED:
7 · Entry-point group

A: a kedro.validators entry-point group for pip-installable adapters in v1  ·  B: v1.x, once a second backend actually exists.

Recommend B  Custom validators already work today via the class path; the entry-point group only changes distribution. Reserve the group name in v1 docs so nothing squats it.

Frame 10 · Risk registerreference — walk only if asked

Known, documented, owned

Top six of the twelve in the KEP. None are blockers; each has a documented mitigation. The theme: where the design has a boundary, we state it loudly rather than let users discover it.

Raw access bypasses validation, by design. catalog.get(name).load() and Viz's preview() skip the funnel — that raw path is exactly what the programmatic API needs to avoid double-validation. Documented contract.

Coercion changes the data. With coerce=True the node and the disk get the coerced frame — a behaviour change when adding a validator to an existing pipeline. Prominent docs and release-notes callout.

Save hooks observe pre-coercion data. The runner holds its own reference, so what's persisted can differ from what after_dataset_saved saw. Documented asymmetry; lineage plugins needing persisted truth read back.

Environment overlays can drop validators. A local override that redeclares an entry without validator: loses it under destructive merge. The CLI reports which validator resolved and from where; docs recommend declaring in base and disabling per-env rather than redeclaring.

Lazy and distributed backends. Load-side validation triggers Spark actions / forces evaluation. Guidance: prefer on: [save] or sampling options for Spark and Dask; per-backend cost table ships in docs.

Generator nodes validate per chunk. Cross-chunk invariants (like unique) are not enforced across the whole output; use on: [load] for appended outputs so the persisted whole is validated on next read.
